# X Account AI Operations — Requirements & Architecture

## Purpose
An AI system that autonomously operates X (Twitter) accounts to build authority, drive engagement, and eventually channel traffic to products.

## Current Account State
- Two accounts managed from a single Linux VPS (Ubuntu 24.04)
- Accounts are in "cold recovery" phase (low posting density to avoid algorithm penalties)
- Both are premium verified
- Goal: 5M impressions / 90 days → platform revenue sharing

## Non-Negotiable Rules
- **Posting density is life and death**: Cold accounts posting high frequency get shadowbanned. Current limits: 2 original posts/day, 8 engagement posts/day, reply-only to @mentions
- **Learning != Posting**: Continuous scanning is 24/7, but the learning agent MUST NEVER post. Posting is handled by separate cron scripts with hard limits
- **Every reply is a profile post**: Gate 0 (evaluate target post quality) + Gate 1 (generate meaningful reply, skip if not good enough) + Gate 2 (profile impact check)

## Current Architecture (All Python 3.12+)

### 1. Continuous Learning Agent (`x_continuous_agent.py`)
- Daemon running 24/7 in a screen session
- Every 1h: quick pulse (check Musk/Altman for breaking news via X API)
- Every 2h: deep analyze (run pulse + viral analyzer)
- Every 6h: self-update strategy (adjust quality gates based on performance data)
- ABSOLUTELY NEVER POSTS

### 2. Viral Mechanics Analyzer (`x_viral_analyzer.py`)
- Studies why posts go viral: hook type (question/quote/statement/news/direct), format (length, media, links), timing
- Tracks our own post performance: impressions, likes, retweets, engagement rate
- Detects breaking news in real-time
- Stores patterns in JSON knowledge base for strategy adjustment

### 3. Reply Worker (`x_ai_worker.py`)
- Daemon running 24/7 in a screen session
- Passive mode: checks @mentions every 30min, evaluates with Gate 0/1 before replying
- Proactive mode: scans mutual followers' recent tweets every 1h, replies to valuable posts (max 2/cycle)
- Uses Gemini 2.5 Flash for reply generation (via API key)

### 4. Original Posting (`x_content_strategy.py`) — Cron: 6/12/18 BJT, max 2/day
- Uses Imagen 4.0 for AI-generated images
- Content generated by MiMo V2 Pro (reasoning model, burns token plan)
- Posts long-form analytical content with images

### 5. Engagement Posting (`x_smart_reply.py`) — Cron: 2/8/14/20 BJT, max 8/day
- Short observations/analysis posts
- Content generated by MiMo V2 Pro
- Posts within same batch spaced 2-5 minutes apart

### 6. Quote-Commentary Engine (`x_quote_commentary.py`) — Cron: 8/14/20 BJT, max 3/day
- Scans 10 accounts (Musk, Altman, Hassabis, LeCun, NVIDIA, AI agents)
- Generates original analysis + embeds original tweet URL
- Traffic flow goes to OUR post, not just a retweet

### 7. Curiosity Module (`x_curiosity.py`) — Cron: every 12h
- Self-discovery: searches X for new accounts/topics/trends to monitor
- Self-critique: reads own performance data, identifies gaps

### 8. Quality Standards (`x_quality_standards.md`)
- Account growth tiers (T3 cold → T2 growth → T1 established → T0 authority)
- Reply quality gates (Gate 0: target post eval, Gate 1: reply substance, Gate 2: profile impact)
- Known failure patterns: topic pool exhaustion, generic content, @-at-start pollution, heartbeat spam, filler replies, topic-locked strategy

## Key Technical Challenges to Solve

### A. Zero-Cost X Access
Currently paying per API call. Ideal solution: Playwright-based scraper that uses browser cookies (exported once per month) to access X.com directly. Challenge: X detects headless Chrome on datacenter IPs and shows captcha/challenge pages.

Attempted solutions that failed:
- Twikit (KEY_BYTE error - X changed login protocol)
- Playwright stealth + cookies (challenge page detected)
- Playwright + xvfb headful (x.com page timeout on VPS)
- Playwright full login flow (selector issues + timeout)

**Needed**: A reliable way to scrape X.com without API costs from a Linux VPS.

### B. Content Quality Improvement
Current content is still too generic. MiMo V2 Pro generates better than Gemini Flash but still produces "mid" content. Need tighter prompt engineering, more specific topic angles, better quality gates.

### C. Cross-Account Learning
@secondary_account (practice) should experiment and feed learnings to @primary_account (main). Currently the pipeline exists but isn't well-optimized.

## Success Metrics
- Engagement rate > 1% (currently 0.6%)
- 5M impressions within 90 days per account
- Post quality: < 20% of posts categorized as "generic/garbage" by self-review
